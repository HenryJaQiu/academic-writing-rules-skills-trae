# 通用学术写作基础规则

本规则面向可跨会议与期刊复用的学术论文项目。它只定义“通用层”约束，不写死某个 venue 的模板、页数、平台字段或当前项目的图号映射。

使用方式：

- `academic_base.md`：跨 venue 通用层
- `NeurIPS.md` 等：venue-specific 叠加层
- `project_rules.md`：当前项目覆盖层

## 1. 主张与证据

- 所有强主张都必须有正文证据承接：定理、实验、消融、权威引用，至少占其一
- 不得把经验现象写成理论保证，不得把存在性结果写成算法性结论
- 若证据只在 controlled setting 下成立，摘要、引言、结论都必须保留边界
- 任何“first / unified / optimal / definitive”类表述都要先经过严格核查

## 2. 引用与事实

- 所有引用必须真实存在，且正文描述与原文贡献一致
- `.bib` 元数据应尽量准确：作者、年份、venue、DOI、arXiv/OpenReview 状态
- Related Work 需要覆盖经典奠基工作与近年关键工作，并说明与本文差异
- 未被正文或附录实际引用的死条目，不应在最终投稿版本中保留

## 3. 结构与写作

- 先明确论文类型：理论、方法、实证、系统、应用或混合型
- 章节顺序要服务读者认知流，而不是只跟着写作时间顺序堆内容
- Introduction 不应承担过多 notation / definitions；形式化准备应放在后续独立节中
- Abstract、Introduction、Conclusion 三处的主线应一致，不能一处强调理论、一处只谈实验、另一处又夸成通用框架

## 4. 图表与可读性

- 图表是证据链的一部分，不是装饰
- legend、tick、标题、标注不得遮挡主要结构，不得被裁切或越界
- 任何修改图内字体、线宽、marker、布局后，都必须同时检查导出的图 PDF 和编译后的最终论文 PDF
- 当用户要求修改特定图时，必须先确认图号、图文件与生成脚本的对应关系；如项目中尚无映射表，应先建立
- 不得靠 LaTeX `trim/clip` 掩盖图本身的排版问题，优先在出图脚本中修复

## 5. 数学与证明

- theorem / lemma / proposition 的正文陈述必须与附录证明口径一致
- 关键 assumptions 应集中定义，并在证明中被明确引用
- 跳步性语言如“clearly”“it is easy to see”后若承载关键逻辑，必须补出细节
- 若某命题在当前假设下无法成立，应收紧 statement 或补充假设，而不是硬写成“可证”

## 6. 迭代与回滚

- 每轮修改优先只处理一个高杠杆问题，并验证副作用
- 修改后要检查是否引入新的矛盾：图文不一致、引用错配、页数变化、交叉引用失效、claim 变强
- 如果一个改动让论文更花哨但不更可信，应回滚

## 7. 编译与交付

- 只有在用户明确要求时才做全文编译；否则优先更新必要产物
- 一旦执行终稿编译，应检查日志中的 undefined references、citation warnings、overfull/underfull 风险及页数信息
- 投稿前要明确区分：内容层问题、venue 层问题、项目局部问题，避免混改

## 8. 可复用维护约定

- 通用规则不写死某个会议/期刊年份细则
- venue-specific 要求单独放入 venue 规则文件
- 项目特有的图号映射、目录结构、实验节奏单独放入 project 规则文件
